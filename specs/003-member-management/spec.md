# Feature Specification: 會員管理頁面

**Feature Branch**: `003-member-management` **Created**: 2026-04-22 **Status**: Draft **Input**: User description: "新增會員管理頁面，一個會員會有所屬的代理商，admin 這個帳號不參與任何會員相關的事項，只用來管理。"

## 背景與角色說明

本系統存在兩種角色：

- **系統管理員（admin）**: 唯一的超級管理員，負責系統層級的維護。可查看並管理所有會員資料，但本身不屬於任何代理商，也不代表任何業務關係。
- **代理商（Agent）**: 業務實體，每位會員必須歸屬於某一個代理商。

---

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 查看會員列表 (Priority: P1)

系統管理員登入後，可在會員管理頁中查看所有會員資料。每筆記錄顯示會員的所屬代理商，並支援分頁瀏覽與條件搜尋。

**Why this priority**: 會員列表是會員管理的基礎視圖，後續所有操作（新增、篩選）皆以此為起點。

**Independent Test**: 登入後訪問會員管理頁，可看到含代理商欄位的會員資料表格（含分頁），即視為可獨立測試的功能切片。

**Acceptance Scenarios**:

1. **Given** 管理員已登入，**When** 訪問會員管理頁，**Then** 頁面顯示會員資料表格，欄位包含：會員 ID、使用者名稱、Email、所屬代理商、狀態、建立時間、操作。
2. **Given** 會員列表頁，**When** 系統載入資料，**Then** 資料依建立時間降冪排列，每頁預設顯示 20 筆。
3. **Given** 會員列表頁，**When** 管理員輸入會員名稱或 Email 關鍵字並執行搜尋，**Then** 表格僅顯示符合關鍵字的會員，分頁重置至第一頁。
4. **Given** 會員列表頁，**When** 管理員從代理商篩選選單選擇特定代理商，**Then** 表格僅顯示屬於該代理商的會員。
5. **Given** 會員列表有多頁資料，**When** 管理員切換頁碼，**Then** 表格載入對應頁的資料並顯示總筆數。
6. **Given** 資料載入中，**When** 資料尚未回傳，**Then** 表格顯示載入中狀態，防止使用者誤判為無資料。

### User Story 2 - 新增會員 (Priority: P1)

管理員點擊「新增會員」按鈕，系統彈出表單對話框，填寫會員資料（含指定所屬代理商）後送出，系統建立新會員並刷新列表。

**Why this priority**: 新增會員是會員管理的核心 CRUD 操作，且必須明確指定所屬代理商，確保資料歸屬正確。

**Independent Test**: 點擊「新增會員」，填寫資料並選擇代理商後送出，列表出現新會員記錄，即視為功能完整可測試。

**Acceptance Scenarios**:

1. **Given** 管理員在會員列表頁，**When** 點擊「新增會員」按鈕，**Then** 系統彈出對話框，包含：使用者名稱、Email、密碼、所屬代理商（選單）欄位。
2. **Given** 對話框已開啟，**When** 所屬代理商選單展開，**Then** 列出所有狀態為啟用中的代理商供選擇。
3. **Given** 對話框已開啟，**When** 填寫所有必填欄位並選擇代理商後點擊「確認」，**Then** 系統建立新會員、關閉對話框、刷新會員列表。
4. **Given** 對話框已開啟，**When** 未填寫必填欄位或未選擇代理商直接點擊「確認」，**Then** 顯示對應欄位的驗證錯誤提示，不送出請求。
5. **Given** 管理員正在送出請求，**When** 系統尚未回應，**Then** 確認按鈕進入不可點擊狀態，防止重複送出。
6. **Given** 管理員送出請求，**When** 系統回應錯誤（如 Email 已被使用），**Then** 顯示具體錯誤提示，對話框保持開啟讓管理員修正。
7. **Given** 對話框已開啟，**When** 點擊「取消」或關閉按鈕，**Then** 對話框關閉，列表不刷新，已填寫內容不保留。

---

### Edge Cases

- 系統中尚無任何啟用中的代理商時，新增會員對話框的代理商選單顯示「暫無可用代理商」提示，確認按鈕不可操作。
- Email 格式不符時，系統即時驗證並提示格式錯誤，不等待送出後才回饋。
- Email 已被其他會員使用時，系統顯示「此 Email 已被使用」提示。
- 代理商選單資料載入失敗時，顯示錯誤提示並提供重試入口。
- 搜尋欄清空後執行搜尋，系統重置篩選條件並載入完整列表。
- 管理員帳號本身不出現在代理商選單中，不可作為任何會員的所屬代理商。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 提供會員管理頁面，列表欄位包含：會員 ID、使用者名稱、Email、所屬代理商名稱、狀態、建立時間、操作。
- **FR-002**: 會員列表 MUST 支援分頁瀏覽，每頁預設顯示 20 筆，並顯示總筆數。
- **FR-003**: 會員列表 MUST 支援依使用者名稱或 Email 進行關鍵字搜尋。
- **FR-004**: 會員列表 MUST 支援依所屬代理商進行篩選（選單篩選）。
- **FR-005**: 系統 MUST 在會員列表頁提供「新增會員」按鈕，點擊後開啟對話框（不跳頁）。
- **FR-006**: 新增會員對話框 MUST 包含以下必填欄位：
  - 使用者名稱
  - Email（需符合電子郵件格式）
  - 密碼（至少 8 個字元）
  - 所屬代理商（選單，僅列出狀態為啟用中的代理商）
- **FR-007**: 系統 MUST 在送出前進行表單驗證，所有必填欄位不得為空。
- **FR-008**: 新增成功後，系統 MUST 關閉對話框並重新載入會員列表（刷新至第一頁）。
- **FR-009**: 送出請求期間確認按鈕 MUST 進入不可點擊狀態，防止重複操作。
- **FR-010**: 系統回應錯誤時，MUST 顯示對應錯誤提示，對話框保持開啟。
- **FR-011**: 系統側邊導覽 MUST 新增「會員管理」入口，可連結至會員列表頁。
- **FR-012**: 僅系統管理員（admin）可存取會員管理頁面，其他角色不得進入。

### Key Entities

- **會員（Member）**: 系統管理的業務實體；屬性：`id`、`username`（使用者名稱）、`email`、`agentId`（所屬代理商，必填不得為空）、`agentName`（所屬代理商顯示名稱）、`status`（啟用 / 停用）、`createdAt`（建立時間）。
- **新增會員請求（CreateMemberRequest）**: 新增操作的輸入資料；屬性：`username`、`email`、`password`、`agentId`。
- **代理商（Agent）**: 同第一期 PRD 定義；`agentId` 為會員的必要歸屬欄位，admin 帳號不可作為代理商使用。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 管理員可在 60 秒內完成新增會員操作（從點擊按鈕到列表顯示新資料）。
- **SC-002**: 表單驗證錯誤在點擊確認後即時顯示（不超過 200ms），不需等待系統回應。
- **SC-003**: 新增成功後列表刷新完成時間不超過 2 秒（系統回應後計算）。
- **SC-004**: 所有系統錯誤（請求失敗、網路中斷）皆有對應的使用者可見提示，不出現無回饋的靜默失敗。
- **SC-005**: 確認按鈕在請求期間 100% 進入不可點擊狀態，防止重複送出。
- **SC-006**: 代理商篩選選單在開啟時 1 秒內完成資料載入（排除網路延遲）。

---

## Assumptions

- 新增會員的初始狀態固定為「啟用中」，不由管理員指定。
- admin 帳號不屬於任何代理商，也不可出現在代理商選單中。
- 本期不包含編輯會員資料與停用/刪除會員功能。
- 本期不涉及代理商端自行管理會員的場景，所有操作僅由 admin 執行。
- 密碼由系統加密儲存，前端不儲存或明文顯示密碼。
- 對話框關閉即丟棄未送出的資料，不需草稿儲存功能。
- 代理商選單資料在對話框開啟時即時載入，以確保選項為最新狀態。
